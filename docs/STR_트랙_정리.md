# STR 트랙 정리 (담당: 상진)

> 신고 구조화(STR) 트랙 전체 작업 정리 문서. **계속 갱신하는 살아있는 문서**다.
> 티켓 완료/수정 시 해당 절과 아래 "진행 현황"·"파일 사전"을 함께 업데이트할 것.
> 최종 갱신: 2026-07-10

---

## 0. 트랙 개요

NHTSA 소비자 신고(자유 텍스트)를 LLM으로 정형 데이터(JSON)로 바꾸는 것이 STR 트랙이다.
- **입력**: 신고 원문(CDESCR) — 영어 자유 텍스트
- **출력**: 구조화 JSON (부품/증상/심각도/상황/근거인용/정보부족여부 + v3부터 리콜언급)
- **LLM**: Gemini 2.5 Flash (temperature=0, JSON 강제). PROVIDER 환경변수로 교체 가능(gemini|anthropic).
- **프롬프트**: 검증된 v2 → 필드 추가한 v3. 프롬프트 본문은 `docs/struct_prompt_v*.md`.

## 2. 티켓별 상세

### STR-01 — 배치 구조화 API
- **한 것**: 신고 CSV를 프롬프트 v2로 LLM 일괄 호출 → 건당 JSON 생성하는 배치 스크립트 작성·실행. 100건 처리.
- **AC 결과**: 형식 위반 0/100 ✅, 배치 100건 완주 ✅.
- **방식**: 건별 [LLM 호출 → JSON 파싱 → 스키마 검사 → 인용 검사 → 실패 시 재시도(최대 3회) → 통과 건만 저장]. 이어하기(처리된 ODINO 스킵)·rate limit 백오프 내장.
- **겪은 것**: 무료 티어 쿼터로 완주 불가 → 유료 결제 후 완주. 1차 5건 실패(모지바케·part_category 괄호 복사) → 스크립트 보강 후 100/100.
- **의미**: 이 스크립트는 100건 전용이 아니라 범용 도구. 입력만 바꿔 전체 1,579건에도 그대로 재사용(`--input`).

### STR-02 — 인용(환각) 자동 검사
- **한 것**: 별도 티켓이 아니라 STR-01 루프에 내장. `evidence_quote`가 원문(CDESCR)에 실제 존재하는지 문자열 대조(공백·따옴표 정규화 후). 불일치 시 파기·재시도.
- **AC 결과**: 인용 통과율 100%(불일치 0/100) ✅. 불일치 건 파기·재시도 동작 확인(1차 실행 때 실제로 작동).
- **의미**: LLM이 근거를 지어내지 않았음을 기계적으로 보장. STR-01 산출물이 "형식뿐 아니라 근거도 신뢰 가능"함을 증명.

### STR-03 — 확장 정확도 측정  ⬜ 미착수
- **할 것**: 정답 라벨을 만들고, 그 정답 대비 100건(v3 최종본)을 채점. strict/relaxed 정확도 + 장문/단문 분리 보고(AC1) + 오류 유형 4분류 집계(AC2).
- **주의**: 신고 구조화의 정답 라벨은 레포에 없음(직접 확인). 정답 만들기부터가 이 티켓. 사람 판정 필요.
- **정본 채점 대상**: `str01_sample100_results_v3.jsonl` (8필드).

### STR-04 — 스키마 v3 (mentions_existing_recall)
- **한 것**: v2를 복사해 v3 작성, `mentions_existing_recall`(bool) 필드 추가(v2는 무수정 보존). 제네시스 14건으로 필드 작동 검증.
- **AC 결과**: 제네시스 대조군에서 리콜 언급 11/11 탐지(오탐 0) ✅ — AC 문턱(9건 이상) 충족.
- **회귀**: 같은 모델로 v2 vs v3 비교, 기존 4필드 56개 중 6개만 상이(경계 케이스, 일부는 temperature=0 실행 노이즈).
- **의미**: 급증 분석 시 "기존 리콜 후속 항의"를 신규 시그널과 분리하는 필드. severity·part_category와 독립.

### (파생) v3로 100건 최종본 생성
- **한 것**: 검증된 v3 프롬프트로 100건 전체를 재구조화 → 8필드 최종 JSON. 방식·모델·입력은 STR-01과 동일, 프롬프트만 v2→v3.
- **결과**: 100/100, 형식 위반 0·인용 불일치 0. mentions_existing_recall true 19 / false 81.
- **정본 정리**: v2 파일(7필드)은 STR-01/02 증빙으로 보존, **v3 파일(8필드)이 이후 단계 정본**.

### STR-05 — 사용자 입력 구조화 (차종·증상 추출)  🟡 커밋 대기
- **한 것**: 한국어 사용자 입력에서 **차종·증상 2개만** 추출하는 모듈. 차종은 carRegistry 27종 영문 표기로 정규화(웹앱 조회와 매칭), 증상은 STR-01 symptoms 형식.
- **되물음(슬롯 기반 A)**: LLM은 한 턴 추출, 코드가 슬롯 보관·병합·상태 확정·루프. 차종/증상 중 부족한 것만 콕 집어 되물어 둘 다 찰 때까지 반복. 완료 시 확인문장 출력.
- **AC**: 차종 미인식 시 되물음(AC1) ✅, 추출 결과 확인 문장 제공(AC2) ✅.
- **테스트**: 기본 6케이스 + 확장 10케이스(여러 차종 모호/증상 여러개/심한 오타/영어차종+한글증상/영어대문자/잡담/구형별칭 K3→FORTE/구어체 은어/단어만/부품만 여러턴) 실행. 등록 밖 차종·여러 차종 모호·잡담은 억지 매핑 없이 되물음, 구어체("갑자기 훅 꺼져요")를 STR 표준 증상어("주행 중 시동 꺼짐")로 정규화 확인.
- **관찰(개선 여지)**: 증상 판정이 다소 관대 — 부품명 단독("브레이크")·모호 표현("계기판 문제")도 증상으로 인정. STR 데이터 매칭 정확도를 높이려면 구체적 증상을 되묻도록 엄격화 검토 가능(현재는 사용자 편의 우선). 트레이드오프: 엄격화 시 되물음 증가.
- **웹앱 연동 지점**: `scripts/cht01_chat_pipeline.py`의 `parse_question`(규칙 기반, "TODO: LLM_CALL 교체 지점") 자리에 `extract_slots()`를 꽂으면 됨. 실제 연동은 웹앱 트랙 작업(범위 밖).
