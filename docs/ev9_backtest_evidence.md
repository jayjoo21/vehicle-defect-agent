# EV9 계기판 리콜(24V757000) 되감기 재현 근거 — 데이터 리키지 방어용

## 3줄 요약

- **리콜 접수일(컷오프)**: 2024-10-10 (NHTSA `24V757000`, `ReportReceivedDate`="10/10/2024" → `report_date_iso`="2024-10-10")
- **신호 발화일**: 2024-09 (월 집계 기준, B1 baseline이 EV9 모델에서 최초로 스파이크 규칙을 만족한 달)
- **선행일**: **10일** (컷오프 2024-10-10 − 발화월 말일 2024-09-30 = 10일). 이 값은 기존 산출물(`b1_eval_k12.csv`)과 이번 재현 실행 양쪽에서 동일하게 확인됨.

## [1] 컷오프·선행일 재계산

### 리콜 접수일 소스 확인
`data/recalls/recalls_hk_by_vehicle.csv`와 `data/recalls/recalls_sw_candidates.csv` 양쪽에서 `24V757000` 행을 직접 조회:

```
NHTSACampaignNumber: 24V757000
Make: KIA / Model: EV9
ReportReceivedDate: 10/10/2024
report_date_iso: 2024-10-10
Component: ELECTRICAL SYSTEM: INSTRUMENT CLUSTER/PANEL
```

두 소스 모두 `report_date_iso = 2024-10-10`으로 일치.

### 신호 발화일 확인
`data/processed/b1_eval_k12.csv`에서 `campaign == 24V757000` 행:

```
models: EV9
report_date_iso: 2024-10-10
window_start: 2023-10 / window_end: 2024-09
captured: True
trigger_model: EV9
first_signal_month: 2024-09
lead_days: 10
window_trend: ... EV9 2024-08 count=1 base=2.0; EV9 2024-09 count=52 base=2.0
```

### 선행일 재계산 (코드로 직접)
```
cutoff = 2024-10-10
first_signal_month = 2024-09  →  월말일 = 2024-09-30
lead_days = (cutoff - 월말일).days = 10
```
`b1_eval_k12.py`의 계산 로직(`(cutoff - first["month"].end_time.normalize()).days`)을 그대로 재실행한 결과도 **10**으로 일치. 임의로 만든 값이 아니라 코드가 실제로 산출하는 값 그대로임.

## [2] 미래 정보 차단 증거

### (a) 감지에 들어간 EV9 신고들의 접수일이 전부 컷오프 이전인지
`data/processed/hk_electrical_recent_full.csv`에서 `MODELTXT == EV9`이고 `LDATE`가 2024-09월인 행을 추출:

- **원본 행 수(LDATE 2024-09월)**: 52건 — `b1_signals.csv`의 `count=52`와 정확히 일치
- **고유 ODINO 수**: 29건 (CLAUDE.md에 이미 기록된 "원본52행 vs 고유29건" 구조와 일치 — 동일 ODINO가 COMPDESC별 여러 행을 갖는 데이터 구조 때문. count는 원본 행 기준이라는 점은 기존에도 명시돼 있던 한계)
- **LDATE 범위**: 2024-09-06 ~ 2024-09-29 → **전부 컷오프(2024-10-10) 이전**, 컷오프 이후 LDATE 0건
- **FAILDATE(고장 발생일) 범위**: 2024-08-05 ~ 2024-09-28 → 결측 0건, **전부 컷오프 이전**, 컷오프 이후 FAILDATE 0건

즉 신호를 발화시킨 52건의 신고는 접수일 기준으로도 고장발생일 기준으로도 리콜 접수일보다 뒤에 있는 것이 하나도 없음.

### (b) 감지 스크립트가 사용하는 입력 컬럼
`scripts/b1_detect.py` 소스 확인 (수정 없음, 그대로 인용):
```python
IN_PATH = "data/processed/hk_electrical_recent_full.csv"
...
df = pd.read_csv(path, encoding="utf-8-sig", dtype=str, usecols=["MODELTXT", "LDATE"])
```
`usecols=["MODELTXT", "LDATE"]` — **차종명과 접수일 단 2개 컬럼만 로드**. `COMPDESC`(부품 라벨)·`CDESCR`(서술문)·리콜 정답 데이터 어느 것도 이 스크립트에 로드되지 않음.

### (c) 부품 지식/리콜 정답 데이터가 감지 입력에 안 쓰였다는 코드 경로 확인
```
grep -inE "rcl573|recall|compdesc|part_number|supplier" scripts/b1_detect.py
→ 매치 0건
```
`b1_detect.py` 전체(69줄)에 `rcl573_*`, `recalls_*`, `COMPDESC`, 부품/공급사 관련 문자열이 **단 한 번도 등장하지 않음**. 리콜 정답 데이터(`recalls_hk_by_vehicle.csv` 등)는 오직 채점 스크립트 `b1_eval_k12.py`에서만 로드되며, 그 용도는 "컷오프 날짜"와 "채점 대상 모델 목록"을 정하는 것뿐 — 탐지 로직(`detect_signals()`)에는 전달되지 않음.

### (d) 구조적으로도 미래 데이터를 볼 수 없는 이유
`detect_signals()`의 베이스라인 계산:
```python
grid["baseline_avg6"] = grid.groupby("model")["count"].transform(
    lambda s: s.shift(1).rolling(6, min_periods=1).mean()
)
```
`shift(1)`이 있어 해당 월의 베이스라인은 **그 달을 포함하지 않는, 그 이전 최대 6개월**만 사용. 즉 `b1_signals.csv`를 만드는 스크립트 자체가 2022~2026 전체 기간 데이터를 한 번에 읽어 실행되었더라도(실제로 그렇게 실행됨), 특정 월(예: 2024-09)의 발화 여부 판정은 산술적으로 그 이후 달의 값을 전혀 참조하지 않음. 아래 [3]에서는 이걸 이론이 아니라 실제로 미래 행을 물리적으로 제거한 뒤 재실행해 증명함.

## [3] 재현 로그

컷오프 이전 데이터만으로 `b1_detect.py`와 동일한 로직(코드 무수정, 로직만 재사용)을 재실행:

```
실행 시각(run_time):      2026-07-22T20:52:23+09:00
입력 파일 경로:            data/processed/hk_electrical_recent_full.csv
입력 파일 SHA-256:         7df5c35a8d838ed7faa4ae31571baf9ef46018bdafd99db654f5b484a858b1e5
원본 전체 행 수(모델+LDATE만): 16,964
LDATE 파싱 실패(NaT):       0건
컷오프(2024-10-10) 이후로 제외된 행 수: 8,028건 (전 차종 대상, EV9 한정 아님)
컷오프 이전으로 남은 행 수:  8,936건
```

입력 파일에서 VIN(식별) 컬럼을 공개 전 제거했으며, 제거 후 감지 결과(발화월·건수·선행일·포착여부)는 제거 전과 동일함을 재확인하였다.

컷오프 이전 8,936건만으로 EV9 모델·2024-01~09월 그리드를 재계산한 결과:

| model | month   | count | baseline_avg6 | signal |
|-------|---------|-------|----------------|--------|
| EV9   | 2024-01 | 0     | 0.00           | False  |
| EV9   | 2024-02 | 1     | 0.00           | False  |
| EV9   | 2024-03 | 0     | 0.17           | False  |
| EV9   | 2024-04 | 9     | 0.17           | False  |
| EV9   | 2024-05 | 1     | 1.67           | False  |
| EV9   | 2024-06 | 1     | 1.83           | False  |
| EV9   | 2024-07 | 0     | 2.00           | False  |
| EV9   | 2024-08 | 1     | 2.00           | False  |
| **EV9** | **2024-09** | **52** | **2.00** | **True** |

```
발화 여부: True
최초 발화월: 2024-09
count: 52 / baseline_avg6: 2.0
lead_days (재현 재계산): 10
```

**원본 산출물(`b1_eval_k12.csv`, 2026-07-06 생성)의 값(발화월 2024-09 / count 52 / baseline 2.0 / lead_days 10)과 이번 재현 결과가 완전히 일치**함. 즉 컷오프 이후 데이터를 8,028건 물리적으로 제거해도 EV9의 감지 결과는 조금도 달라지지 않음.

> **리콜 정답(24V757000) 사용 범위 명시**: 위 재현 실행에서 리콜 데이터(`recalls_hk_by_vehicle.csv`)는 컷오프 날짜(2024-10-10)를 정하는 데만 사용되었고, 탐지 로직 자체(그리드 구성·베이스라인 계산·스파이크 판정)의 입력으로는 전혀 사용되지 않았다 — 탐지는 `MODELTXT`·`LDATE` 두 컬럼만으로 수행되며, 채점(정답 대조)만 리콜 데이터가 관여한다.

## [4] 한계 명시

- **부분 재현(Partial Reproduction)**이지 완전 재현이 아님. 이번 검증은 "현재(2026-07) 시점에 보관 중인 `hk_electrical_recent_full.csv`를 `LDATE < 컷오프` 조건으로 걸러낸 것"이지, "2024-10-10 당일 NHTSA가 실제로 공개하고 있던 데이터의 스냅샷"을 그대로 보존해서 쓴 것이 아니다. 이 프로젝트는 2024-10-10 시점의 원본 스냅샷을 별도로 보관해두지 않았으므로, 아래 전제가 성립해야 이 재현이 "당시 실제로 볼 수 있었던 데이터"와 동일하다:
  - NHTSA 불만 레코드가 접수(LDATE) 이후 수정되거나 삭제되지 않는다(사후 편집으로 LDATE가 과거인 레코드가 나중에 새로 나타나지 않는다).
  - NHTSA가 신고를 접수 즉시(지연 없이) 공개 데이터셋에 반영한다.
  - 이 두 전제 자체는 **이 데이터셋 하나만으로는 검증 불가** — "확인 불가"로 명시함. NHTSA 측의 데이터 게시 지연(레코드가 실제 접수일보다 며칠~몇 주 늦게 공개 데이터에 반영되는 경우)이 실무에서 존재할 수 있다는 점은 별도 조사가 필요.
- `count`(52)는 원본 CSV의 행 단위 집계이며 고유 신고 건수(29건, ODINO 기준)보다 많다 — 이는 리키지와 무관한 기존에 이미 알려진 데이터 구조 특성(CLAUDE.md에 기록됨)이지만, 발표 시 "52건 신고"라고 말하면 부정확하므로 "원본 행 기준 52건(고유 신고 29건)"으로 정확히 표현할 것.
- 이번 재현은 EV9 1개 캠페인(24V757000)에 대해서만 수행함. K12 전체 12개 캠페인에 대한 동일한 컷오프-절단 재현은 이번 범위 밖(요청 시 동일 방법으로 확장 가능).

## 논문에 실을 수 있는 문장 초안

> 본 연구의 급증 탐지 baseline(B1)은 차종·월 단위로 집계된 신고 건수와 그 접수일(LDATE)만을 입력으로 사용하며, 부품 지식이나 리콜 정답 데이터는 탐지 로직에 관여하지 않는다. 기아 EV9 계기판 리콜(NHTSA 캠페인 24V757000, 접수일 2024-10-10)에 대해 리콜 접수일 이전 데이터만으로 탐지 파이프라인을 재실행한 결과, 2024년 10월 10일 이후에 접수된 8,028건(전 차종 대상)을 완전히 제외하고도 동일하게 2024년 9월에 신호가 발화했으며(원본 행 기준 52건, baseline 대비 26배), 리콜 접수일 대비 10일의 선행 탐지가 재현되었다. 이는 탐지 결과가 리콜 발표 이후 공개된 정보(부품 원인 문서, 정답 리콜 데이터)에 의존하지 않았음을 실증한다. 다만 이 재현은 현재 보관 중인 데이터셋을 접수일 기준으로 사후 절단한 것이며, 리콜 발표 시점 당시 NHTSA가 실제로 공개하고 있던 데이터의 원본 스냅샷을 대조한 것은 아니라는 점에서 완전한 시점 고정 재현은 아니다.
