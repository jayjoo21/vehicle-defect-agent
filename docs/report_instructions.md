# 프로젝트 보고서 작성 지시서 (Claude Code용)

## 목표
MOBISCOPE 프로젝트의 A-to-Z 기록 보고서를 LaTeX(article)로 작성한다. **논문 형식 + 보고서 내용**: 구조·조판은 학술 논문처럼 격식 있게, 문체는 formal하되(줄글, "~이다/~했다" 체, "우리는" 허용), 내용은 결과 자랑이 아니라 **가설→시도→실패→개선→교훈의 여정**을 담는다.

## 절대 규칙 (환각 방지 — 최우선)
1. **모든 수치·파일 경로·실험 결과는 저장소 실측에서만 가져온다.** 기억으로 숫자를 쓰지 말 것. 확인 안 되는 값은 `[확인 필요]`로 표시하고 넘어간다.
2. **참고문헌은 저장소/노션에 실제로 근거가 있는 것만 BibTeX에 넣는다.** 논문 제목·저자·연도를 지어내지 말 것. 근거 파일이 없으면 `references.bib`에 `% TODO: 서지정보 확인 필요` 주석으로 자리만 만들고 본문에서는 \cite 없이 서술.
3. 각 실험 수치를 쓸 때 출처 파일을 LaTeX 주석(`% 출처: data/processed/xxx.csv`)으로 남긴다.

## 사전 작업 (본문 쓰기 전)
1. 저장소에서 다음을 읽어 수치 대장(fact sheet)을 먼저 만든다: CLAUDE.md, docs/ 전체(특히 05·06·08·09·10·11 문서), data/processed의 주요 결과 CSV, str01_sample100_v4 결과, str01_sample20_v4_grading.
2. 참고문헌 근거를 찾는다: 노션 export, docs/, README 등에서 논문 인용·URL·서지정보가 있는 파일을 검색. 발견한 것만 references.bib로.
3. 그림 후보를 조사한다: 기존에 만든 SVG·차트·스크린샷(webapp 대시보드/채팅/내차, 모델 배치도, 아키텍처 다이어그램)이 저장소에 있으면 목록화. 없으면 TikZ/표로 대체.

## 문서 구조 (LaTeX article, 국문)

**Title / Authors / Abstract** — Abstract는 문제·방법·핵심결과(되감기 10/12, 환각 0, UNKNOWN 26/27 구제 등)·기여를 200자 내외로.

**1. 서론**
- 문제: 리콜은 사후적이고 늦다 / 차량 SW·전장 결함 증가 추세(11문서 통계: 미국 23%, 한국 42~52%, 자체 46% 삼각검증)
- 목표와 3대 설계 원칙(폐쇄 근거 / 모르면 모른다 / 진단 아님 고지)
- 기여 3가지 명시: ① 되감기(backtesting) 기반 객관 평가 프로토콜 ② 신뢰 불가 라벨 데이터에서 COMPDESC 앵커링 우회 ③ 가설 기각까지 포함한 정직한 실험 방법론

**2. 배경: 좋은 AI Agent의 조건과 본 주제의 적합성**
- LLM vs Agent 구분(노션 3장), 좋은 Agent 6기준(노션 4장: 역할분리/동적경로/불확실성통제/자가검증/추적성/평가가능성)
- 본 주제가 6기준을 어떻게 충족하는지 대응표
- 왜 LLM이 필수인가: UNKNOWN OR OTHER 34%(1,702건)에 심각 신호가 묻힘 → 라벨 집계로는 불가, 원문 독해 필요

**3. 데이터**
- 수집: NHTSA 불만 222만 → 현대·기아·전장·2022+ 필터 → 16,964건. 미국 리콜 164(SW후보 76), 한국 리콜(보도자료+KOTSA 10년치)
- 입력 데이터 vs 정답지 데이터 분리 개념(문제지 vs 채점표)
- K12 되감기 문제셋 구축(유형 A/B/C)
- **데이터 함정 기록(정직 서술)**: 49 vs 51 필드, 차종명 정규화(HYBRID 접미어), 한·미 매칭 오염, 우측 절단 — 각각 어떻게 발견·처리했는지

**4. 시스템 설계**
- 파이프라인 6단계: 감지(B1, 통계) → 구조화(LLM) → 조사 루프 → 판정(Self-Consistency) → 검수(Judge) → 리포트. 다이어그램
- 각 단계가 수업 개념의 무엇을 구현하는지 매핑(Structured Output, ICL, Test-Time Scaling, Verifier Judge)
- 모델 라우팅(Hybrid): 감지=코드, 구조화=경량(Gemini 2.5 Flash), 조사·판정=고성능, Judge=교차 벤더
- v4 구조화의 COMPDESC 앵커링: 부품을 LLM이 재분류하지 않고 원본 라벨을 관점 지정용 입력으로 사용 + UNKNOWN 전용 형제 규칙

**5. 실험 (핵심 장 — 여정 서술)**
각 실험을 "가설 → 방법 → 결과 → 판정 → 배운 것" 소절 구조로. 10_실험로그.md 기반.
- 5.1 B1 감지 baseline: 포착 10/12, 그러나 우연 대비 리프트 +24%p·정밀도로 검증하니 "감지만으론 결함 규명 불가" → LLM 필요성 역증명
- 5.2 구조화 미니테스트: 20건 strict 90% → **심각도 과대평가라는 체계적 편향 발견** → 판정규칙 3종(ISO 26262 Controllability 등) 주입한 v2 → 20/20 교정. (실패→개선의 대표 사례)
- 5.3 집중도 가설: n=2에서 유망 → **n=10에서 기각**(0.645 vs 0.661) → "리콜 판정은 단일 지표 아닌 다기준" 역증명. (정직한 실패 기록)
- 5.4 한·미 매칭 오염: 그럴듯한 "한국선행 다수" 관측 → 대표 사례 눈검증서 오매칭 발견 → 방향 통계는 검증된 27건만 신뢰로 후퇴
- 5.5 v4 전환·100건 검증: 환각 0/100, 형식위반 0, **UNKNOWN 27건 중 26건 구제**(형제 8 + 본문 18), insufficient 1. 스키마 없이 UNKNOWN 문제 해결

**6. 평가 방법론**
- 되감기(backtesting) 프로토콜: 리콜 이전 데이터만 제공→포착·선행일·근거정확성 채점. 우연 포착 대조(null model) 병기
- 골드라벨 교차 채점: 5인 독립 채점→다수결 정답, IAA로 Judge 신뢰도 검증(진행 중). 채점 대상=severity·symptoms·insufficient(부품은 원본이라 제외)
- 2층 평가: 모듈 평가(사람) vs 시스템 평가(되감기, 객관)

**7. 웹 서비스 구현**
- 3화면(상황판/조사채팅/내차 3D) + 데모 시나리오(EV6→투싼 연결). 목 모드로 키 없이 동작하는 어댑터 구조. 스크린샷 있으면 삽입

**8. 한계와 향후 과제**
- 정직 서술: 일부 모듈 미통합(cht01↔inv 인터페이스 불일치), 정밀도는 하한 추정, 한·미 정밀 매칭은 LLM 과제로 로드맵, 로컬 sLLM은 chosen/rejected 로그 축적 후 향후
- 협업에서 배운 것: 인터페이스 계약의 필요성(통합 시 발견), 문서-기반 소통

**9. 결론** — 기여 3가지 재확인 + "완벽한 agent보다 완결된 baseline"이라는 수업 철학의 실천

**참고문헌** — references.bib, 실제 근거 있는 것만.
